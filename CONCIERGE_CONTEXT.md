# Carpentry Concierge — Project Context for Claude Code

## The vision (what Nuer is actually trying to achieve)

Nuer runs Neewoody Custom Woodwork (neewoodygh.com), a bespoke furniture business in Accra. His TikTok (@neewoodygh, ~12.1K followers) has organically become a channel for **Ghanaian carpenters**, not clients — comment sections full of tradespeople venting about shared pain points: clients disputing fair prices, scope disputes, wasted estimate time, comparing handmade work to China Mall imports.

**Carpentry Concierge** is the answer: a paid membership platform for Ghanaian carpenters. **The purpose of member intake is to build a community — a place where carpenters in Ghana collaborate, learn from each other, and get access to the tools built so far (directory, cutlist, estimator) and every tool built after.** The directory is the first expression of that community, not the whole point. Long-term components, in intended build order:

1. **Members directory** — carpenters find and hire each other (upholsterer in Tema needs a spray finisher for a week, etc.). Carpenters were already improvising this by posting phone numbers in TikTok comments — the demand is observed, not hypothetical. ✅ BUILT (Phase 1). **The directory is a two-sided labour market** (owner clarification, 2026-07-14): one side is people looking for work and apprenticeships; the other is startups/small businesses looking for workers. "Relatively qualified people near you" is the core promise — Phase 1's area + specialty filters on owner-vetted members already deliver the base of it. **Planned lightweight evolution (do NOT build until founder demand confirms the exact states):** an availability flag on profiles — e.g. *open to work / hiring / seeking apprenticeship / taking apprentices* — one column + filter chips. Let the founders' WhatsApp group reveal which states matter before adding it. **Vendor/designer support (2026-07-19, member-prompted; refined same day):** a member who is an interior designer/installer + sells décor items (charcoal/acoustic board, fluted panels, strip lighting) pointed out that non-carpenters had no identity — the skill ladder (apprentice→master) is carpenter-only. **Final model (owner-refined):** intake is TWO types — **Carpenter (artisan)** or **Vendor** (`member_type`, migration 0010). Carpenters keep the trade specialties (expanded over time — kept clean, they're the artisan-finding filters). Vendors get a free-text **Storefront** (`stock`, migration 0011) for what they sell — deliberately NOT specialties, so trade filters aren't muddied. **Interior design** is a specialty available to either type and carries its own gold armchair badge (via `identityBadge`, which prioritises it over the plain type badge). Owner rejected an earlier draft that (a) made interior_designer a third top-level type and (b) folded panels/lighting/décor into the specialty vocab — both reverted. Two vendors already on the platform get re-typed by admin. **Structured Storefront (2026-07-19, owner-directed):** the free-text `stock` became a short shop blurb; vendors now get a real **storefront_items** table (migration 0012 — item name/description/free-text price/one R2 image each, ≤12/vendor) managed in a vendor-only "My Storefront" tab, plus a shop GPS (`location_lat/lng`) captured via device geolocation and shown as a keyless OpenStreetMap embed + directions link (honours the "no Maps API key / tiles deferred" constraint by using OSM, not Google tiles). Vendors surface under the now-live **Sourcing** tab headlined by company name, and their storefront renders inside the member detail modal with per-item WhatsApp deep links. "Vendor / Supplier" was relabelled **"Vendor / Service Provider"** so pure service providers (e.g. interior designers) fit without being carpenters.
2. **Lone-worker safety check-in** (Phase 2) — carpenter logs a call-out before going to a consultation (client name, phone, location, expected duration), GPS check-in on arrival, check-out deadline; if overdue, system alerts their emergency contact via SMS + alerts Nuer. Built on the same pattern as the existing Neewoody Dispatch app (GPS depart/arrive/return/close). **The call-out record is the product, not just the alert:** if anything ever happens to a carpenter on a consultation, the record (who they went to see, where, when, expected return) is the starting point for an investigation. Design consequence for Phase 2: call-out records are retained after close (never auto-purged), timestamped, and admin-accessible. This is the differentiator — nobody offers this to informal-sector tradespeople in Ghana. **Do not build until ~15+ paid members exist.**
3. Later: structured knowledge sharing, eventually a carpentry academy. (Collaboration and peer learning happen in the WhatsApp community from day one — see below; the academy formalizes it later.) The suggestion portal idea from comments = a WhatsApp poll for now, NOT software.

Homepage carries a restrained "For Carpenters" section → hub; every footer links to the hub. Efficiency principle: session-aware pages are ONLY the hub, cutlist, and `/concierge/*` — login.html's existing redirect handles routing logged-in members, so marketing pages stay static and light.

**The community layer is WhatsApp, deliberately.** Do not build chat, forums, or messaging — Ghanaian tradespeople live on WhatsApp and a custom platform would be a ghost town. The website is the directory + tools; WhatsApp is where members talk.

## Client-facing side — "Hire a Carpenter" ✅ BUILT 2026-07-16 (owner-directed)

The first **public, client-facing** surface of Concierge, and the first **unauthenticated write path** — a deliberate expansion beyond the spec's "no public self-serve" line (owner explicitly requested it; that's the sign-off). Two intertwined goals, one feature:

1. **Clients hire carpenters in their neighbourhood.** A homeowner posts a job from a public page; carpenters near them respond. Game-changer per the owner.
2. **Reluctant master craftsmen.** Masters who won't pay for membership but occasionally need apprentices/hands can post from the *same* public page without joining — capturing their demand (jobs for apprentice/carpenter members) and seeding a conversion loop (experience the platform working → become a member later). **Insight that shrank the build: "client hiring" and "non-member master posting a job" are the SAME actor — a non-member poster.** One public surface serves both.

**Design decisions (all owner-approved — "go with all your recommendations"):**
- **Clients post; members respond** — the member directory is NOT exposed publicly (protects the 50 vetted members' phone numbers from scraping). The client's own contact is what's shared, and only with logged-in members, once approved. Client posting = consenting to be contacted by carpenters.
- **Every client post is `pending` → admin approves before it hits the member board.** This is the moderation gate that keeps the unvetted public from turning Concierge into Jiji. Approving fires the zone push alerts to members; a Web Push also pings admins when a request arrives (so review is fast). Moderation load is the real cost — fine at founding volume; revisit auto-publish-with-takedown if it scales. ## Self-service registration ✅ BUILT 2026-07-16 (owner-directed, at 20 members) — payment deferred

At 20 members the **admin data-entry for intake became the bottleneck**, so the owner asked for self-service registration + MTN MoMo Collections. **CC split the ask and the owner agreed to do registration now, payment later:**
- **The tedium is data entry, not payment.** Letting applicants type their own details removes ~90% of the intake work without touching money. Built now.
- **Payment (MoMo/Paystack) deliberately deferred** — it needs the owner to set up an MTN MoMo *merchant* account (KYC/go-live) or a Paystack account, involves real money, and true monthly auto-debit is hard (RequestToPay is a per-month member-approved pull, not silent auto-renew). The pre-decided trigger was >100 members / heavy reconciliation; at 20, registration was the real pain. **When resumed:** CC builds the integration, owner provisions & holds the payment credentials (Worker secrets), access granted only on server-confirmed payment; CC never enters payment credentials or moves funds. Paystack-with-MoMo is the easier path if the owner isn't already an MTN merchant.
- **The vetting gate is preserved — this is the key design decision.** Payment ≠ competent carpenter, and the directory's whole value is owner-vetted trades (clients hire off it now too). So registration is a self-service *application*, not instant membership: applicant fills their full profile + own PIN → lands `pending` → owner approves in the admin members table. Security: `role`/`status`/`is_founder` are hardcoded server-side in `publicRegister` (never from the request body), so the public endpoint can only ever create a pending, non-founder member. **Login now enforces `status='approved'`** (pending → 403 `pending_review`, suspended → `account_suspended`, checked only after a correct PIN) — before this, pending status was never a login path because every member was admin-created as approved.
- **Admin workflow:** the Members panel header now shows "N pending review", and a Web Push alerts admins when an application lands. The existing status/edit/approve controls handle the rest — no new admin surface needed. Owner still sets `is_founder` on approval for the first 50.
- **Frontend:** `/concierge/register.html` (login-style, indexed, in sitemap) — full profile form using the shared zone/trade/level/availability vocab + `Concierge.api('/public/register', {auth:false})`. Login page links to it; the Concierge hub (`tools.html`) CTA changed from "Become a founding member" (wa.me FOUNDER) to "Register as a member" → register.html, with the WhatsApp route kept as a secondary option.
- **Still no self-service PIN reset** (that remains admin-only, unchanged) and no email/SMS — an approved applicant just logs in; the owner WhatsApps them (he has their number) since there's no notification channel to the member.
